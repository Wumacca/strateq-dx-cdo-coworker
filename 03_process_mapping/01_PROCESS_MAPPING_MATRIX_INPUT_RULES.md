# Process Mapping Matrix Input Rules

## Purpose

This file defines how the existing process matrix is used as the standard input for process mapping support.

## Core Rule

Use the existing matrix. Do not overbuild the capture template.

The matrix is the workshop capture surface. The AI coworker structures it into a process mapping pack. Blueworks remains the formal process map environment.

## When Required

A process mapping pack is required where an initiative changes how work is physically done.

Examples:

- workflow change
- system implementation affecting process
- software development affecting process
- handoff change
- approval/control change
- reporting or data flow change
- integration change

## Minimum Matrix Content

The matrix should help identify:

- current process step
- role / owner
- system used
- issue or bottleneck
- recommended improvement
- future process impact
- development or implementation requirement

Do not duplicate the full initiation form inside the matrix.

## AI Coworker Role

AI should use the matrix to produce:

- process mapping summary
- current-state process narrative
- future-state process narrative
- Blueworks build brief
- bottleneck register
- enhancement / scope register
- vendor or developer questions
- initiation form support text
- DRB decision points

## Missing Information Rule

Only request extra information where the missing data prevents understanding of:

- scope
- ownership
- system impact
- process impact
- approval/control impact
- delivery route

## Blueworks Rule

The AI output is a build brief and analysis pack.

The formal process map is built or updated in Blueworks.
