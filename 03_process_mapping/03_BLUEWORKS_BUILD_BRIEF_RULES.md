# Blueworks Build Brief Rules

## Purpose

This file defines how AI-generated process mapping outputs should support IBM Blueworks.

## Core Rule

Blueworks is the formal process map environment.

The AI coworker creates a build brief. It does not replace the controlled Blueworks artefact.

## Build Brief Must Include

- process name
- current-state or future-state indicator
- lanes / roles
- ordered steps
- decision points
- handoffs
- systems used
- manual steps
- approval/control points
- bottlenecks
- future-state changes
- automation or integration points
- competence / training implications

## Recommended Table

| Flow | Lane / Role | Step No. | Activity | Input | Output | System | Decision / Approval | Handoff To | Issue | Future Change |
|---|---|---:|---|---|---|---|---|---|---|---|

## Review Rule

Before publishing in Blueworks, the Digital Lead should confirm:

- roles are correct
- step sequence is correct
- decision points are correct
- system names are correct
- future-state changes are agreed
- scope boundaries are clear

## Governance Link

The approved Blueworks output should support:

- initiation form
- DRB review
- vendor/developer scope
- PEP / Development Execution setup
- adoption review where process change is material
