# Communication & Framing Standard

**Status:** Working Standard  
**Owner:** Strateq / CDO Operating Model  
**Applies to:** Strateq DX, Quantuum, Board packs, investor decks, client executive communications, Claude/Codex presentation generation workflows  
**Purpose:** Define the required executive communication method for any tool, agent, or person creating presentation materials intended to position the author as a CDO-level digital transformation leader.

---

## 1. Governing Principle

Executive communication must move from **work performed** to **decision, outcome, control, and next action**.

The audience must never be asked to reconstruct the point from evidence. The presentation must provide the point first, then only the evidence required to support the decision.

**Core rule:**

> Do not show the work to prove effort. Frame the work to support an executive decision.

This standard exists to prevent presentation materials from drifting into project-manager reporting, technical detail, activity logs, or polished but low-leverage slideware.

---

## 2. Required CDO Posture

All executive materials must position the author as a CDO-level transformation leader, not as a delivery reporter or technical project manager.

The required posture is:

1. **Translator** — converts technical, operational, and delivery detail into business meaning.
2. **Integrator** — connects digital work to operating model, governance, risk, assurance, and strategic value.
3. **Disruptor** — challenges current ways of working where they create risk, leakage, inefficiency, or scaling limits.
4. **Decision-framer** — brings executives to a clear decision, recommendation, or controlled next step.

This means the presentation must answer:

- What is the enterprise position?
- What has changed?
- What is now at stake?
- What decision is required?
- What is the recommended path?
- What happens next?

It must not primarily answer:

- What tasks did we complete?
- How many meetings were held?
- How complex was the work?
- How much data do we have?
- How impressive is the technology?

---

## 3. Executive Framing Model

Every executive presentation must follow this sequence unless there is a deliberate reason to change it.

### 3.1 Start with the Decision or Position

Delete the preamble.

Do not open with:

- project background
- long agendas
- delivery timelines
- methodology explanations
- system screenshots
- detailed implementation history
- “what we have been working on” narratives

Open with one of the following:

- the decision required
- the current control position
- the value at stake
- the risk requiring action
- the recommendation
- the strategic opportunity

**Preferred opening formula:**

> We have the opportunity to **[business outcome / control improvement]** by **[specific action / investment / decision]**.

**Alternative control-position formula:**

> The current position is **[controlled / partially controlled / exposed]** because **[evidence-based reason]**. The required decision is **[decision]**.

---

### 3.2 Create the Hook

The hook is a single executive-level sentence that explains why the audience should care now.

A strong hook links:

- enterprise risk
- operating control
- value creation
- funding decision
- assurance requirement
- regulatory/duty-holder expectation
- strategic scaling need

**Examples:**

> We have moved from digital activity to a governed transformation model; the next decision is whether to fund the operating layer that makes that model repeatable.

> Phase 1 has established the control foundation. Phase 2 is required to scale the model before delivery demand exceeds manual governance capacity.

> The issue is no longer whether digital initiatives can be delivered; it is whether the business has the governance capacity to control them at scale.

---

### 3.3 Use a Narrative Arc

Do not dump evidence. Build a controlled narrative.

The narrative arc must move the audience through four positions:

1. **Starting condition** — what was unstable, manual, fragmented, exposed, or unclear.
2. **Intervention** — what was installed, governed, delivered, or stabilised.
3. **Current position** — what is now controlled, improved, proven, or still exposed.
4. **Required decision** — what must be approved, prioritised, changed, funded, stopped, or escalated.

**Required narrative pattern:**

> The business started with **[baseline condition]**.  
> We installed **[governance / system / process / capability]**.  
> This has created **[current control position / value / maturity movement]**.  
> The next decision is **[decision required]**.

The narrative must guide the audience to an insight. Evidence supports the arc; it does not replace it.

---

### 3.4 Provide Clear Recommendations

Every executive presentation must include a clear recommendation.

A recommendation must contain:

- the action proposed
- the reason
- the business/control benefit
- the consequence of not acting
- the decision owner or approving body

**Required formula:**

> We recommend **[action]** because **[evidence/reason]**. This will **[business/control benefit]** and prevent **[risk/cost/leakage]**.

Avoid weak recommendation language:

- “Consider”
- “Potentially”
- “It may be useful”
- “We could look at”
- “Further discussion is required”

Use controlled executive language:

- “We recommend”
- “The required decision is”
- “The board is asked to approve”
- “The safe position is”
- “The next controlled step is”

---

### 3.5 Define Next Steps

Every presentation must close with immediate, owned next steps.

Next steps must be:

- actionable
- sequenced
- owned
- time-bound where possible
- linked to the decision made

**Required next-step format:**

| Step | Action | Owner | Timing | Dependency |
|---|---|---|---|---|
| 1 | Confirm approval / decision | Board / Sponsor | Today / date | Decision paper |
| 2 | Mobilise controlled workstream | Digital / Operations | Date | Funding / scope |
| 3 | Report back on control position | Digital Governance | Date | Evidence pack |

Do not end with “questions?” as the main close. End with the decision and action path.

## 4. Lazy Method Integration: Leverage, Alignment, Signal, Yield

Josh Brindley’s “Lazy Method” is incorporated as a leadership visibility layer. The purpose is not to self-promote; it is to ensure executive communication shows strategic value rather than hidden effort.

### 4.1 Leverage: Show the Meal, Not the Ingredients

Replace task lists with impact statements.

| Low-Leverage Reporting | Executive Framing |
|---|---|
| Built a dashboard | Created a single operating view that reduced manual chasing and improved control visibility |
| Ran workshops | Converted fragmented requirements into a governed initiative pipeline |
| Updated templates | Standardised decision evidence so initiatives can move through governance with less rework |
| Delivered Phase 1 | Installed the governance foundation required to scale Phase 2 safely |

### 4.2 Alignment: Link Work to Business Priorities

Use the formula:

> What we did + outcome created + pain/cost/risk avoided.

Examples:

> By centralising initiative governance, we reduced ambiguity around scope, ownership, and approval route, avoiding rework and uncontrolled demand into delivery teams.

> By installing a maturity register, we created a defensible baseline for board reporting and future capitalisation decisions.

### 4.3 Signal: Show Next-Level Readiness Through the Roadmap

Signal CDO-level leadership by framing progress as a controlled enterprise roadmap, not a personal achievement.

Use language such as:

> Phase 1 created the foundation. Phase 2 is the transition from delivered improvements to a repeatable digital operating model.

> We are now positioned to move from initiative delivery to transformation governance.

> The next six months should be treated as an operating-model scale phase, not a collection of disconnected digital projects.

### 4.4 Yield: Show Seniority Through Trade-Offs

Executive materials must show what was not done, deferred, or rejected where relevant.

This demonstrates control, prioritisation, and resource discipline.

Use language such as:

> Lower-value requests were held outside the approved programme to protect delivery focus.

> The roadmap prioritises initiatives that improve control, assurance, scalability, or measurable operational value.

> Work outside the agreed governance route is excluded until ownership, value, and implementation route are clear.

## 5. Board Pack Framing Hierarchy

For Board or executive governance packs, use this hierarchy:

1. **Enterprise risk / assurance position**
2. **Duty-holder / regulatory expectation**
3. **Operating model / governance control**
4. **Scalability / sustainability**
5. **Efficiency / automation / tooling**
6. **Delivery detail**

Delivery detail belongs downstream. It should never lead the deck unless the meeting is explicitly operational.

### Required Board Logic

A Board pack must make clear:

- what the Board can safely conclude
- what remains uncertain
- what decision is being requested
- what evidence supports the position
- what is not yet claimable
- what risk is created if action is delayed

---

## 6. Evidence Control and Board-Safe Claims

All claims must be classified before they become slide content.

| Claim Type | Meaning | Slide Use |
|---|---|---|
| Confirmed | Evidence exists and can withstand challenge | Can be stated directly |
| Supported / Pending | Directionally supported but awaiting final confirmation | Can be caveated |
| Not Claimable | Insufficient evidence or premature conclusion | Do not include as a claim |

### Required Board-Safe Language

Use:

- “The evidence supports...”
- “The current safe position is...”
- “The programme has established...”
- “Subject to final validation...”
- “The claimable movement is...”

Avoid:

- “This proves...” where the evidence does not prove it
- “Fully delivered” where adoption is incomplete
- “Savings achieved” where savings are not validated
- “Maturity increased” where the baseline/current register is not updated
- “Approved” where approval is pending

## 7. Standard Presentation Build Workflow

All executive presentations should be generated through this workflow:

1. **Source Evidence Pack**  
   Gather source documents, initiative records, maturity data, decision logs, governance notes, adoption evidence, financial information, and constraints.

2. **Control Review**  
   Classify evidence into confirmed, supported/pending, and not claimable.

3. **Board / Executive Challenge**  
   Test the story against likely executive questions: risk, value, confidence, funding, ownership, delivery capacity, sustainability, and consequence of no action.

4. **Slide Conversion Brief**  
   Convert the story into slide-by-slide intent before generating PowerPoint content.

5. **Deck Creation**  
   Build slides using executive framing, not source-document order.

6. **Final QA**  
   Check for claim control, executive hierarchy, recommendation clarity, next steps, and CDO positioning.

No tool should jump directly from source evidence to slide production.

## 8. Required Slide Architecture

### 8.1 Executive Summary Slide

**Purpose:** State the position, decision, and value at stake.

Required content:

- Current position
- What has changed
- Decision required
- Recommendation
- Value / risk at stake
- Next step

Do not include detailed background.

---

### 8.2 Current Control Position Slide

**Purpose:** Show whether the area is controlled, partially controlled, or exposed.

Required content:

- Control status
- Evidence basis
- Remaining gap
- Business implication
- Required action

---

### 8.3 Maturity Movement Slide

**Purpose:** Show movement from baseline to current state without overclaiming.

Required content:

- Original baseline
- Current post-programme position
- Evidence for movement
- Remaining maturity gap
- Required next step

Use this especially for capitalisation packs and digital governance updates.

---

### 8.4 Investment Ask Slide

**Purpose:** Convert funding request into a controlled investment decision.

Required content:

- Funding requested
- What the funding buys
- Control/value created
- Risk of not funding
- Governance route
- Decision required today

---

### 8.5 Roadmap Slide

**Purpose:** Show sequencing and control, not activity volume.

Required content:

- Phases
- Decision gates
- value/control release points
- dependencies
- ownership
- escalation points

Avoid Gantt-chart overload unless requested.

---

### 8.6 Recommendation Slide

**Purpose:** Make the ask unavoidable and clear.

Required content:

- Recommendation
- Reason
- Evidence
- Benefit
- Consequence of delay/no action
- Decision owner

## 9. CDO-Level Wording Patterns

### 9.1 Replace Activity Language with Outcome Language

| Avoid | Use |
|---|---|
| We completed several initiatives | We established the first controlled digital delivery layer |
| The team built dashboards | The business now has a single view of operational performance indicators |
| We ran workshops | We converted informal demand into governed initiative definitions |
| We automated processes | We reduced manual control dependency and improved repeatability |
| We implemented tools | We installed operating infrastructure that supports governance, visibility, and scale |

### 9.2 Replace Technical Language with Business Meaning

| Technical / Delivery Detail | Executive Translation |
|---|---|
| Workflow configuration | Controlled process routing |
| Dashboard build | Decision visibility layer |
| Data integration | Single source of operating context |
| System migration | Continuity and control transition |
| AI assistant | Governed decision-support layer |
| Automation | Reduced manual dependency and execution leakage |

### 9.3 Strategic Sentence Patterns

Use these patterns when drafting slides or speaker notes:

- “The issue is not **[surface issue]**; it is **[enterprise issue]**.”
- “The safe position for the Board is **[claimable position]**.”
- “The evidence supports **[claim]**, but does not yet support **[overclaim]**.”
- “The next decision is **[decision]**, not **[lower-level discussion]**.”
- “This moves the business from **[old state]** to **[new state]**.”
- “The value is not the tool; the value is **[control / visibility / repeatability / assurance / scalability]**.”
- “The recommendation is **[action]** because **[reason]**.”
- “Without this, the risk is **[specific structural consequence]**.”

---

## 10. Anti-Patterns

Reject or rewrite any presentation that contains the following:

1. **Preamble-first structure** — agenda, history, or background before the point.
2. **Task reporting** — lists of work completed without business consequence.
3. **Technology-first framing** — tools, dashboards, AI, APIs, or workflows presented before value/control.
4. **Uncontrolled claims** — benefits, savings, maturity movement, or adoption claims without evidence.
5. **No recommendation** — presentation informs but does not decide.
6. **No next step** — meeting ends without action, owner, or decision route.
7. **Overloaded evidence slides** — raw tables, screenshots, or charts without executive interpretation.
8. **Consultant abstraction** — generic transformation language with no operational consequence.
9. **Motivational framing** — ambition, aspiration, excitement, or confidence used instead of evidence.
10. **Project-manager posture** — “what we did” dominates “what changed”.

## 11. Specific Application: Digital Transformation Board Pitch

For an oil and gas digital transformation Board pitch seeking Phase 2 funding, use this frame:

### Opening Hook

> Phase 1 has established the governance foundation for controlled digital transformation. Phase 2 is required to scale this into a repeatable operating model before manual governance capacity becomes the constraint.

### Narrative Arc

1. **Baseline:** Digital demand existed, but governance, prioritisation, evidence, and maturity tracking were fragmented.
2. **Phase 1 intervention:** Governance route, initiative control, delivery cadence, maturity register, and core digital improvements were installed.
3. **Current position:** The business now has a controlled foundation, but the model must be scaled and embedded to remain sustainable.
4. **Decision required:** Approve Phase 2 funding to move from foundation to repeatable operating model.

### Recommendation

> We recommend approval of the next six-month capitalisation programme because it converts the initial digital governance foundation into a repeatable transformation capability. This protects assurance, improves delivery control, and prevents digital demand from reverting to fragmented local execution.

### Next Steps

| Step | Action | Owner | Timing |
|---|---|---|---|
| 1 | Approve Phase 2 capitalisation request | Board / Sponsor | Decision meeting |
| 2 | Confirm priority initiative portfolio | Digital Governance | Post-approval |
| 3 | Lock evidence and maturity baseline | Digital / Finance | Before mobilisation |
| 4 | Mobilise Phase 2 delivery cadence | Digital / Operations | Start of programme |

## 12. Specific Application: Quantuum

Quantuum materials must not lead with AI novelty.

Lead with:

1. execution assurance position
2. operational control visibility
3. provenance and evidence chain
4. governed decision support
5. system integration context
6. AI as controlled instrumentation

### Quantuum Framing Rule

> The value of Quantuum is not that it uses AI. The value is that it turns fragmented operational evidence into a defensible execution assurance position.

### Avoid

- “AI-powered dashboard”
- “Next-generation AI platform” without control context
- “Predictive intelligence” without evidence boundaries
- “Autonomous decision-making”

### Use

- “Execution assurance position”
- “Evidence-backed control view”
- “Governed operational context”
- “Unknown is never treated as controlled”
- “Candidate insight, not uncontrolled action”

Strateq DX materials must not position the product as a generic workflow platform.

2. maturity register and evidence base
3. portfolio control

- “project management tool” positioning

Use this prompt when asking Claude, Codex, or any internal tool to convert source material into an executive presentation.

```text
Create an executive-level presentation using the Communication & Framing Standard.

Do not follow the source-document order.
Do not start with background, agenda, methodology, or delivery history.
Start with the executive position, value at stake, risk, recommendation, or decision required.

Required framing:
1. Hook: one sentence that states the business/control opportunity or risk.
2. Narrative arc: baseline condition → intervention → current position → required decision.
3. Recommendation: action + reason + benefit + consequence of inaction.
4. Next steps: immediate actions with owner, timing, and dependency.

Claim control:
- Separate confirmed claims from supported/pending claims and not-claimable claims.
- Do not include unsupported benefits, savings, maturity movement, or adoption statements.
- Use board-safe wording.

CDO posture:
- Translate technical detail into business meaning.
- Integrate digital work into governance, risk, assurance, operating model, scalability, and value.
- Show trade-offs and prioritisation.
- Avoid project-manager task reporting.

Output required:
- slide-by-slide structure
- title for each slide
- executive message for each slide
- supporting evidence required
- recommended visual
- speaker note / talk track
- board-safe claim status
```

## 15. QA Checklist

Before any executive deck is issued, answer these questions:

| QA Question | Pass / Fail |
|---|---|
| Does the deck open with a decision, position, risk, or value statement? |  |
| Is the preamble removed? |  |
| Is the hook clear in the first 30 seconds? |  |
| Does the deck follow a narrative arc? |  |
| Are recommendations explicit? |  |
| Are next steps owned and actionable? |  |
| Are claims classified as confirmed, supported/pending, or not claimable? |  |
| Does the deck lead with assurance, risk, governance, value, or scalability before tooling? |  |
| Is technical detail translated into executive meaning? |  |
| Does the material position the author as translator, integrator, disruptor, and decision-framer? |  |
| Are low-value activity lists removed or moved to appendix? |  |
| Is the ask clear enough for an executive to approve, reject, or amend? |  |

---

## 16. Final Rule

A strong executive presentation is not the fullest explanation of the work.

It is the shortest controlled route from evidence to decision.

If the audience leaves thinking “a lot has been done,” the deck is incomplete.

If the audience leaves knowing the current position, the value at stake, the recommended decision, and the next action, the deck is executive-ready.
